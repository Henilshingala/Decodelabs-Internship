## Overview

A supervised machine learning project that classifies flower species from
four numeric measurements (`sepal_length`, `sepal_width`, `petal_length`,
`petal_width`) using a **K-Nearest Neighbors (KNN)** classifier built with
scikit-learn.

## Objectives

- Build a complete, reproducible ML classification pipeline: load → preprocess → scale → split → train → predict → evaluate.
- Demonstrate proper feature scaling for a distance-based algorithm.
- Report and interpret accuracy, precision, recall, F1-score, and a confusion matrix.

## Problem Statement

Given four flower measurements, predict which of three species
(`setosa`, `versicolor`, `virginica`) the sample belongs to. This is a
classic multi-class classification problem, ideal for demonstrating KNN.

## Theory

KNN classifies a sample by majority vote among its `K` nearest neighbors in
feature space, measured by Euclidean distance. Because distance is sensitive
to feature scale, **`StandardScaler`** is applied before training. See
[`model_training.md`](model_training.md) for the full theoretical walkthrough.

## Implementation & Code Explanation

`classification.py` is organized into small, testable functions:

| Function | Purpose |
|---|---|
| `load_dataset()` | Reads CSV, validates existence/non-emptiness |
| `preprocess_data()` | Drops NaNs, splits features/target, label-encodes species |
| `scale_features()` | Fits `StandardScaler` on train, transforms both train/test |
| `train_knn()` | Fits `KNeighborsClassifier(n_neighbors=5)` |
| `evaluate_model()` | Computes accuracy/precision/recall/F1/confusion matrix |
| `plot_confusion_matrix()` | Renders and saves the confusion matrix heatmap |

## Advantages

- Simple, interpretable algorithm with no training phase (lazy learner)
- Performs well on small, clean, low-dimensional datasets
- Easy to explain to non-technical stakeholders

## Limitations

- Computationally expensive at prediction time on large datasets (must compare to all training points)
- Sensitive to irrelevant/noisy features and the choice of `K`
- Performance degrades in high-dimensional feature spaces ("curse of dimensionality")

## Common Mistakes

1. **Scaling after splitting incorrectly** — fitting the scaler on the entire dataset (including test data) leaks test-set information into training. Always `fit` the scaler only on training data.
2. **Forgetting `stratify=` in `train_test_split`** — can produce an unbalanced test set for small datasets.
3. **Using accuracy alone** — misleading on imbalanced datasets; always check precision/recall/F1 per class.
4. **Choosing K=1** — almost always overfits to noise.
5. **Not encoding categorical labels** before fitting scikit-learn classifiers, which expect numeric targets.

## Interview Questions

1. **Why is feature scaling necessary for KNN but not for, say, Decision Trees?**
   KNN relies on distance calculations, where unscaled features with larger numeric ranges dominate the distance metric. Decision Trees split on thresholds per feature independently and are scale-invariant.

2. **How do you choose the value of K?**
   Via cross-validation, trying a range of odd values (to avoid ties) and selecting the one that minimizes validation error — balancing bias (too large K) and variance (too small K).

3. **What does macro-averaged F1 mean versus weighted F1?**
   Macro-average treats every class equally regardless of size; weighted average weights each class's score by its number of true samples (support). Macro is preferred for balanced datasets when every class matters equally.

4. **Why fit the scaler only on the training set?**
   To avoid data leakage — the test set must remain unseen during any parameter-fitting step (including scaling) to give an unbiased estimate of generalization performance.

5. **What is the time complexity concern with KNN at inference time?**
   Naive KNN computes the distance to every training point for each prediction — O(n) per query, which becomes a bottleneck for large datasets unless optimized with structures like KD-Trees or Ball Trees.
